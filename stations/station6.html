AI excels at

  • Generating basic animation syntax: AI writes clean transition and keyframes code quickly when you give it specific requirements
  • Creating keyframes boilerplate: Multi-step animations with proper percentage markers and structure are AI's specialty
  • Converting timing values between units: Need to convert 300ms to seconds or understand frame rates? AI handles conversions instantly
  • Explaining animation concepts clearly: AI breaks down complex topics like cubic-bezier curves and animation-fill-mode with excellent clarity
  • Suggesting multiple approaches: Ask for animation solutions and AI offers several valid techniques to choose from
  • Generating starting code quickly: AI creates functional boilerplate in seconds, giving you a foundation to build on

AI struggles with

  • Remembering to include prefers-reduced-motion: AI constantly forgets accessibility unless explicitly prompted every single time

  • Choosing appropriate timing for context: Error notifications need different timing than success messages ‚Äî AI makes generic choices
  • Optimizing for mobile performance: AI doesn't prioritize GPU properties unless you specifically request them
  • Making semantic naming choices: AI generates names like animation1 instead of meaningful names like slideInNotification

Optimizing animation performance with AI

Performance is another area where AI needs explicit guidance. AI knows about GPU-accelerated properties in theory, but it doesn't prioritize them unless you specifically request optimization. I learned this when I built an animation that worked beautifully on my desktop computer but stuttered badly on Prof. Teeters' phone. AI had used CPU-bound properties like margin and width instead of GPU-accelerated transform and opacity. The lesson: AI can help optimize performance, but only if you tell it performance is a priority.

Remembering the GPU "fast lane" from Station 2

In Station 2, Prof. Teeters taught me about the GPU "fast lane" for transform and opacity. These properties get special graphics-card acceleration that makes animations silky smooth, even on slower devices. When I animate width, margin, or position properties instead, the browser has to recalculate the entire page layout on every frame ‚Äî and that stutters on mobile! Now I needed to teach AI to follow the same rule.

The challenge: Animation stuttering on mobile

I created a card animation that expanded when clicked. My first version worked perfectly on my laptop ‚Äî smooth, responsive, professional. Then I tested it on Prof. Teeters' older phone and it was janky! The animation stuttered and felt sluggish. I checked the code and realized the problem: I'd asked AI for a "card expansion animation" without specifying performance requirements. AI generated code using width and height properties (CPU-bound) instead of transform: scale (GPU-accelerated).

The performance-focused prompt

Here's how I learned to request performance optimization explicitly:

‚úÖ Performance optimization prompt:
Optimize this animation for mobile performance with these requirements:

Performance constraints:
- Use ONLY transform and opacity (GPU-accelerated properties)
- NO margin, padding, width, height, top, left, right, bottom (CPU-bound)
- Explain why your choices are performant
- Keep animation duration under 300ms for responsiveness
- Ensure smooth playback on devices with 60fps displays

[Then describe the animation you need...]

Before and after: Performance comparison

Here's the dramatic difference between CPU-bound and GPU-optimized code:

‚ùå CPU-bound animation (stutters on mobile)

/* Card expansion - CPU-bound properties */
.card {
  width: 300px;
  height: 200px;
  margin: 20px;
  transition: width 300ms ease-out, height 300ms ease-out;
}

.card.expanded {
  width: 400px;
  height: 300px;

  /* ‚ùå Animating margin shifts surrounding elements */

}

Why this stutters: Width, height, and margin are layout properties. Changing them forces the browser to recalculate positions of all surrounding elements on every animation frame. This work happens on the CPU's main thread, which gets overwhelmed on mobile devices.

Now here's the GPU-optimized version that AI generates when I request performance explicitly:

‚úÖ GPU-optimized animation (smooth on mobile)
/* Card expansion - GPU-accelerated properties */

  width: 300px;
  height: 200px;
  transform: scale(1);
  opacity: 1;
  transition: transform 300ms ease-out, opacity 300ms ease-out;
}

.card.expanded {
  transform: scale(1.33);
  opacity: 0.95;
  /* ‚úÖ Transform happens on GPU's graphics pipeline */
  /* ‚úÖ No layout recalculation needed */
  /* ‚úÖ Surrounding elements unaffected */
  /* ‚úÖ Smooth on all devices, including older phones */
}

@media (prefers-reduced-motion: reduce) {
  .card {
    transition: none;
  }
  .card.expanded {
    transform: scale(1);
    opacity: 0.9;
  }
}

Why this is smooth: Transform and opacity are GPU-accelerated properties that run on a separate graphics thread. The browser doesn't need to recalculate layout or paint ‚Äî it just composites the pre-rendered element at a different scale. This works smoothly even on low-powered mobile devices!

Ask for mobile-friendly code that runs smoothly on 60fps displays. Performance isn't automatic ‚Äî AI prioritizes "works correctly" over "works smoothly on slow devices." If you don't mention performance, AI will suggest easy properties like margin and width that destroy mobile performance. Make performance constraints explicit!

3

Provide context and purpose

Explain what the animation is for (error notification, success message, loading indicator). Mention your brand personality (playful, professional, energetic). Describe user expectations for this interaction. Give timing preferences based on urgency. Context dramatically improves AI's decisions! When AI knows an error notification should feel urgent (200ms, sharp timing) versus a success message should feel celebratory (400ms, bouncy easing), it makes better choices. Don't just describe the technical requirements ‚Äî explain the "why" behind the animation.

4

Use detailed specific prompts

List all technical requirements upfront (properties, duration, timing function, fill-mode). Include accessibility constraints explicitly. Mention performance needs (GPU properties only). Specify exact approaches you want (use keyframes not transitions, use ease-out not linear). Detailed prompts save time ‚Äî they get you production-ready code on the first attempt instead of code you have to fix. Vague prompts force AI to guess, and its guesses rarely match professional standards. Spend two minutes crafting a comprehensive prompt to save two hours of fixing!

5

Verify everything before using

Read every line of generated code before copying it. Test accessibility manually with DevTools prefers-reduced-motion emulation. Check performance on mobile devices (preferably older phones). Validate against concepts learned in Stations 1-5. Never trust AI output blindly ‚Äî verification is mandatory! AI generates code in seconds, but you're responsible for every line forever. If it breaks, causes accessibility violations, or performs poorly, that's your failure, not AI's. Verification isn't optional.

6

Understand before shipping

You must comprehend every line of AI-generated code. If you don't understand why it works, you can't debug when problems arise. You can't modify code you don't comprehend. Knowledge beats speed ‚Äî shipping code you understand is better than shipping code quickly. Prof. Teeters taught me that understanding is what separates professionals from code copiers. AI can generate animations, but you need to know why the timing works, why ease-out feels right, why transform is performant. That knowledge makes you valuable, not the code itself.

Advanced prompt strategies

Real AI collaboration isn't one-shot perfection ‚Äî it's conversational refinement. I learned that the best results come from treating AI like a colleague I'm working with, not a magic wand I wave once. These advanced prompts help me iterate, troubleshoot, and refine AI output safely. Prof. Teeters calls this "conversational refinement" and it's how professionals actually work with AI.

Iteration prompt for refinement

Use this when the code works but doesn't quite feel right. Maybe the timing is too slow, the easing is too sharp, or the effect is too subtle. This prompt helps you refine while preserving what's already correct:

Refinement prompt:
Revise this animation code. Keep the same technical approach
(transform/opacity, prefers-reduced-motion) but adjust the timing
to feel more [snappy/smooth/gentle] for a [context: e.g., error notification].

Current code:
[paste the code AI generated]

Make it feel more urgent without breaking accessibility or performance.

When to use this: When technical requirements are met but aesthetic feel needs adjustment. Perfect for refining timing and easing to match brand personality while keeping accessibility and performance intact.

Troubleshooting prompt for problems

Use this when something isn't working correctly. AI is surprisingly good at debugging when you describe the specific problem you're experiencing:

Debugging prompt:
This animation doesn't work correctly: [describe issue].

Current code:
[paste the problematic code]

Explain what might be wrong with the current implementation,
then provide a corrected version that maintains accessibility
and performance requirements.

Specific problem: The card doesn't stay in its final position
after the animation completes.

When to use this: When you spot a bug or unexpected behavior. AI will often identify issues like missing fill-mode, incorrect timing functions, or conflicting properties. But always verify the fix yourself ‚Äî understand the root cause!

Safety prompt for production code

Use this when refactoring existing code that's already working in production. This prompt ensures nothing critical breaks during changes:

Safe refactoring prompt:
Refactor this animation without breaking existing functionality.
Preserve all:
- Accessibility features (reduced-motion, keyboard focus)
- Performance optimizations (GPU properties)
- Class names and IDs (JavaScript may reference them)
- Visual appearance (timing, easing, distances)

Current code:
[paste existing working code]

Goal: Convert from transitions to keyframes while maintaining
everything that currently works.

When to use this: When improving or modernizing code that users depend on. Ensures you don't accidentally remove accessibility, break JavaScript dependencies, or change visual behavior unexpectedly.

When NOT to use AI

AI is powerful for animations, but it's not always the right tool. Prof. Teeters taught me to recognize situations where manual work is better than AI assistance. Using AI when you shouldn't is worse than not using AI at all ‚Äî it creates dependencies on tools for things you should understand deeply yourself. Understanding when NOT to use AI is just as important as knowing when to use it.

When you're still learning

Don't use AI for concepts you don't understand yet. If you're still learning how keyframes work, don't let AI write them for you. Do it manually first to build understanding, then use AI for repetition once you comprehend the fundamentals. Prof. Teeters put it perfectly: "You can't verify what you don't understand. You can't debug what you didn't learn. AI should accelerate your knowledge, not replace it."

Practice animations manually through Stations 1-5 first. Once you know why ease-out feels responsive, why transform is performant, and why prefers-reduced-motion is mandatory, then AI becomes valuable for speed. But skip the learning and AI just makes you dependent without building capability.

When timing must match brand

AI doesn't understand brand personality. Should your button hover feel playful and bouncy or professional and subtle? That's a design judgment requiring human intuition. AI makes generic choices ‚Äî it suggests 300ms ease-out for everything because that's statistically common. But your brand might need 200ms linear for urgency, or 500ms ease-in-out for luxury.

Use AI to generate the technical structure, but adjust timing and easing yourself based on brand personality. AI can write the syntax, but you have to provide the soul. Context-specific timing decisions need your judgment, not AI's statistical averages.

When accessibility is critical

Never trust AI for accessibility alone. Always verify reduced motion implementation manually. Check that alternatives work properly. Test with real accessibility tools. Even when you explicitly request accessibility in prompts, AI sometimes gets it wrong ‚Äî missing edge cases, providing incomplete alternatives, or suggesting patterns that don't actually preserve functionality.

AI can generate the @media query syntax, but you must verify it works correctly for users. Test with DevTools emulation. Verify the static alternative makes sense. Ensure functionality is preserved without motion. Accessibility verification is your responsibility ‚Äî AI can assist, but you must validate.
