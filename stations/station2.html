Introduction to CSS transitions

Transitions are the simplest way to add smooth animation to CSS. They automatically animate property values between two states over a specified duration. No JavaScript required, no complex animation libraries ‚Äî just one line of CSS tells the browser to smoothly interpolate values instead of changing them instantly.

What transitions are

Smooth animation between two states

Transitions work by detecting when a CSS property value changes ‚Äî like when you hover over a button and its background color changes. Instead of snapping instantly to the new value, the browser calculates all the in-between values and creates a smooth animation. The starting state and ending state are automatic; you just control the duration and which properties to animate.

How transitions work

Browser automatically calculates in-between values

When a property changes, the browser looks at the old value and the new value, then gradually changes from one to the other over the time you specified. This process is called interpolation. For colors, it blends between shades. For transforms, it smoothly moves through intermediate positions. The browser handles all the math automatically!

Basic transition syntax:
.button {
  background: hsl(32, 76%, 63%);
  transition: background 300ms;
}

.button:hover {
  background: hsl(32, 76%, 53%);
  /* Smoothly transitions from lighter to darker orange */
}

The transition property goes on the base state (not the hover state). This tells the browser: "When background changes, animate it over 300 milliseconds." That's all you need for smooth color changes!

Common use cases for transitions

Transitions work best for simple state changes where you have a clear starting point and ending point. Here are the most common situations where I use them:

Interactive states

  • Button hover effects (color, scale, shadow changes)
  • Link hover states (color, underline appearance)
  • Form input focus states (border color, glow effects)
  • Active states when clicking elements

JavaScript-triggered changes

  • Adding/removing classes that change styles
  • Toggling visibility or display states
  • Opening and closing menu panels
  • Showing and hiding modal dialogs

Duration and property selection

Getting transitions right comes down to two critical decisions: how long should they last, and which properties should animate. I learned both of these lessons through spectacular failures ‚Äî a 2-second button that felt like watching paint dry, and a phone that stuttered because I animated everything at once!

Understanding duration: How long is just right?

Duration controls how long the transition takes from start to finish. You can specify it in milliseconds (ms) or seconds (s), but Prof. Teeters taught me to always use milliseconds for consistency. The duration you choose completely changes how the animation feels!

The sweet spot: 250-500ms

Fast enough to feel responsive, slow enough to see

This range works for most UI animations. Buttons, links, hover states, menu items ‚Äî if you're unsure about duration, start with 300ms. Prof. Teeters calls it the "default duration" because it works in almost every situation. It's fast enough that users don't feel like they're waiting, but slow enough that the motion is visible and smooth.

Too fast: 100ms and under

Barely noticeable on many screens

At 100ms, the transition happens so quickly that users might not even register it as motion. It feels almost instant, which defeats the purpose of smooth transitions. Prof. Teeters explained that animations this fast only work for very subtle effects where you want a hint of motion without drawing attention.

Too slow: 1000ms (1 second) and above

Feels sluggish and frustrating

Anything over 1 second makes users feel like they're waiting for the interface to catch up with them. For UI elements, slow animations communicate "loading" or "processing," not polish. Prof. Teeters has a strict rule: if an animation makes you tap your fingers waiting for it to finish, it's too slow!

Duration comparison examples:
/* Too fast - barely noticeable */
.button-fast {
  transition: background 100ms;
}

/* Sweet spot - responsive and visible */
.button-perfect {
  transition: background 300ms;
}

/* Too slow - feels sluggish */
.button-slow {
  transition: background 1000ms;
}

/* Way too slow - ridiculous for UI */
.button-ridiculous {
  transition: background 2000ms;
}

Prof. Teeters taught me to hover over each of these durations to feel the difference. The 300ms version feels polished and professional. The 2000ms version feels like a joke!

Understanding property selection: Be specific, always

The second critical decision is which properties to transition. This is where I made my biggest mistake ‚Äî using transition: all because it seemed convenient. Prof. Teeters has a strict rule about this with absolutely no exceptions: Never use transition: all.

üî¥ The danger of transition: all

‚ùå WRONG: Never use transition all

.button {
  background: hsl(32, 76%, 63%);
  transform: scale(1);
  transition: all 300ms; /* Animates EVERYTHING that changes */
}

.button:hover {
  background: hsl(32, 76%, 53%);
  transform: scale(1.05);
  /* Only two properties change, but all are being watched */
}

‚úÖ RIGHT: List specific properties

.button {
  background: hsl(32, 76%, 63%);
  transform: scale(1);
  transition: transform 300ms ease-out,
              background 300ms ease-out;
}

.button:hover {
  background: hsl(32, 76%, 53%);
  transform: scale(1.05);
  /* Only the properties we named are animated */
}

Why specific properties matter:

  • Performance: transition: all watches every property for changes, even ones you didn't intend to animate. This creates extra work for the browser.
  • Unexpected animations: Properties like width, margin, padding might change in ways you didn't expect, causing janky motion.
  • Mobile suffers: On slower devices (budget phones), transition: all causes visible stuttering and dropped frames.
  • Debugging nightmare: When something animates unexpectedly, you can't tell which property is causing the problem.

Prof. Teeters taught me that the few extra seconds to list specific properties is always worth the smooth performance you get in return!

Delay, shorthand syntax, and accessibility

Now that we understand duration and property selection, there are three more critical pieces to master: transition delay for choreographing multiple animations, the shorthand syntax for writing transitions efficiently, and the absolutely mandatory accessibility pattern that respects user motion preferences.

Understanding transition delay

Delay controls how long to wait before the transition starts. It's measured in milliseconds just like duration. When I first learned about delay, I completely misunderstood its purpose ‚Äî I thought it was for making users wait. Prof. Teeters had to correct that thinking fast!

What delay is for: Rhythm and choreography

Creating sequential animations, not obstruction

Delay is useful when you want multiple elements to animate in sequence, like menu items appearing one after another with a slight stagger. The delays should be very short (50-100ms) to create rhythm, not long waits. Prof. Teeters taught me that delay is about polish, not punishment!

What delay is NOT for: Making users wait

Never use delay to slow down interactions

I made the mistake of adding 2-second delays to popup notifications because I thought users needed time to "prepare" for them. Prof. Teeters explained that delays like that just frustrate users. If an element needs to appear, it should appear quickly. Delays are for choreography, not obstruction!

Delay syntax examples:
/* No delay - starts immediately */
.button {
  transition: background 300ms ease-out 0ms;
}

/* Small delay - subtle pause before starting */
.button {
  transition: background 300ms ease-out 50ms;
}

/* Staggered menu items - each starts slightly after the previous */
.menu-item {
  opacity: 0;
  transform: translateY(-10px);
  transition: opacity 300ms ease-out,
              transform 300ms ease-out;
}

.menu-item:nth-child(1) { transition-delay: 0ms; }
.menu-item:nth-child(2) { transition-delay: 50ms; }
.menu-item:nth-child(3) { transition-delay: 100ms; }
.menu-item:nth-child(4) { transition-delay: 150ms; }
.menu-item:nth-child(5) { transition-delay: 200ms; }

.menu.open .menu-item {
  opacity: 1;
  transform: translateY(0);
}

The menu items create a cascading effect with 50ms increments. This adds rhythm without making users wait ‚Äî the total stagger is only 200ms from first to last!

Transition shorthand syntax: The PDTD order

You can write transitions in shorthand format by listing all four values in a specific order: Property, Duration, Timing function, Delay. Prof. Teeters taught me to remember this with the mnemonic "PDTD."

Shorthand syntax breakdown:
/* Full shorthand: Property Duration Timing Delay */
.button {
  transition: background 300ms ease-out 0ms;
  /*          ^property  ^dur   ^timing  ^delay */
}

/* Timing and delay are optional (defaults: ease and 0ms) */
.button {
  transition: background 300ms;
  /* Uses default timing function (ease) and no delay */
}

/* Multiple transitions with commas */
.card {
  transition: transform 300ms ease-out,
              background 300ms ease-out,
              opacity 200ms ease-in;
}

Property and duration are required. Timing function and delay are optional ‚Äî if you leave them out, the browser uses defaults (ease timing, 0ms delay).

HAP's PDTD memory trick

I kept mixing up the order of shorthand values ‚Äî sometimes I'd write duration before property, other times I'd put delay before timing. My transitions would either fail silently or behave weirdly, and I couldn't figure out why. Prof. Teeters noticed my confusion and taught me the PDTD mnemonic: Property Duration Timing Delay. She made me repeat it out loud: "P-D-T-D, P-D-T-D." Then she explained that you can remember it like talking to a friend: "This PROPERTY takes this DURATION with this TIMING, starting after this DELAY." üî¥ CRITICAL: Respecting user motion preferences with prefers-reduced-motion

This is the most important part of Station 2. Every transition you create MUST include the prefers-reduced-motion media query. This is not optional, not a nice-to-have, and not something you can skip. It's absolutely required for accessibility.

Why prefers-reduced-motion matters

Some users experience motion sickness, migraines, or vestibular disorders from animations

Prof. Teeters explained that animations we think are subtle and polished can cause real physical discomfort for some users. People with vestibular disorders can experience nausea, dizziness, or headaches from motion on screen. The good news is that operating systems let users set a "reduce motion" preference, and CSS can detect and respect that preference with a media query!

How it works:

  • Users enable "Reduce Motion" in their OS accessibility settings
  • CSS detects this with @media (prefers-reduced-motion: reduce)
  • We turn off transitions with transition: none;
  • Elements still change states ‚Äî they just happen instantly instead of smoothly
  • Functionality is preserved, motion is removed

Prof. Teeters' rule: Station 1 transforms were instant (no motion), so reduced motion was easy. Station 2 introduces MOTION, which means this query is now mandatory for every single transition. No exceptions!

The required pattern (use this EVERY time):
/* Base button with transitions */
.button {
  background: hsl(32, 76%, 63%);
  transform: scale(1);
  transition: transform 300ms ease-out,
              background 300ms ease-out;
}

.button:hover {
  background: hsl(32, 76%, 53%);
  transform: scale(1.05);
}

/* REQUIRED: Respect user motion preferences */
@media (prefers-reduced-motion: reduce) {
  .button {
    transition: none;
    /* Button still changes color and scale */
    /* Changes just happen instantly instead of smoothly */
  }
}

/* Alternative: Turn off ALL transitions globally */
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    transition: none !important;
  }
}

The first approach (targeting specific elements) gives you fine control. The second approach (global wildcard) is simpler and ensures you never miss an element. Prof. Teeters recommends the global approach for most projects!

Testing with Chrome DevTools:

How to test your transitions respect reduced motion:

  1. Open Chrome DevTools (F12 or Cmd+Option+I)
  2. Open Command Menu (Cmd+Shift+P or Ctrl+Shift+P)
  3. Type "Rendering" and select "Show Rendering"
  4. Scroll to "Emulate CSS media feature prefers-reduced-motion"
  5. Select "reduce"
  6. All transitions should disappear instantly
  7. Elements should still work ‚Äî they just change instantly instead of smoothly
  8. Switch back to "no-preference" to see transitions return

Prof. Teeters taught me to test EVERY page with this setting before considering it done. This is part of the workflow, not an optional check!

  • Understand 250 to 500ms range for UI animations
  • Know that faster usually feels better
  • Recognize when animations feel too slow
  • Can use milliseconds correctly

Property selection mastery

  • Can list specific properties to transition
  • Understand why transition: all is problematic
  • Know which properties commonly need transitions
  • Can use comma-separated multiple properties
  • Understand performance implications of property choice

Transition shorthand

  • Can use the shorthand transition syntax
  • Know the order: property duration timing delay
  • Understand when timing and delay are optional
  • Can write multiple transitions with commas
  • Remember PDTD mnemonic for order

Delay understanding

  • Know what transition delay controls
  • Understand delay is for rhythm, not waiting
  • Can use delays for staggered effects
  • Know to keep delays short and subtle
  • Recognize when delay improves choreography

Accessibility compliance

  • Can write prefers-reduced-motion media query
  • Understand why reduced motion matters
  • Know how to test with Chrome DevTools
  • Always include query with every transition
  • Understand that accessibility is mandatory

Performance awareness

  • Know that specific properties perform better than all
  • Understand that some properties are slower to animate
  • Recognize that desktop performance differs from mobile
  • Aware that Station 3 will cover this deeper
  • Ready to learn about GPU versus CPU animation
